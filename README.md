# DerivIQ Analyzer

A measurement instrument for Deriv volatility indices. It reads the public tick
feed, extracts the last digit of every quote, and reports what the sample
actually contains — distribution, uniformity, serial correlation, and whether
the common "hot digit" heuristic beats chance.

It does not predict, and it does not trade.

## Deploying to Vercel

No account linking or CLI is required if you push this to GitHub first.

1. Push this folder to a repository.
2. Go to **vercel.com/new** → Import Git Repository → select it.
3. Vercel auto-detects Vite. Leave the defaults:
   - Build command: `npm run build`
   - Output directory: `dist`
4. Deploy.

There are **no environment variables and no secrets**. Public market data
requires no authentication, so there is nothing to configure and nothing to
leak. This is the main reason the analyzer was separated from anything that
trades.

## Running locally

```bash
npm install
npm run dev      # http://localhost:5173
npm run build    # production build into dist/
```

## Architecture

### Verified behaviour

The statistics are tested, not assumed:

- Chi-square p-values match published critical values for df = 9 to five
  decimal places (16.919 → 0.05000, 21.666 → 0.01000, 3.325 → 0.95001).
- A simulated fair feed of 20,000 digits returns p ≈ 0.49, entropy 3.3216 of a
  maximum 3.3219 bits, and a hot-digit hit rate of 10.4%.
- A feed biased toward one digit is correctly flagged (χ² ≈ 7000, p ≈ 0).
- Last-digit extraction handles trailing zeros: `1234.5670` at 4 decimal places
  yields `0`, not `7`. Getting this wrong silently skews every distribution,
  and most digit apps get it wrong.

### Request format, and how it resolves itself

Deriv has been migrating its WebSocket request format. The Deriv API index
states that "the symbol field is `underlying_symbol`", so that shape is sent
first for both `ticks` and `ticks_history`, with the older string form kept as a
fallback.

If the server rejects a shape, the client advances to the next one
automatically. Requests are tracked **per endpoint** and errors are matched to
the request that caused them via the `echo_req` field Deriv echoes back, so
several in-flight requests cannot be confused with one another.

Whichever format succeeds is named in the **Connection log** panel:

```
accepted: ticks: format 1 of 3 — {"ticks":{"underlying_symbol":"R_100"},"subscribe":1}
```

That line is the answer to "which format does Deriv actually want" — once it
appears, the fallbacks can be deleted. Verified against a simulated server that
rejects the first shape: all three endpoints resolve, and errors are attributed
correctly when history and ticks are in flight simultaneously.

## On what this app claims

Deriv's volatility indices are generated by a cryptographically secure random
number generator. Each digit is independent of every digit before it, so the win
probability of every digit contract is fixed at its mathematical baseline —
roughly 10% for Matches, 90% for Differs, 50% for Even/Odd. No model, filter,
confidence threshold, entry-timing scheme or staking plan changes those numbers.

The **Predictability check** panel exists to demonstrate this rather than assert
it: it continuously backtests the hottest-digit heuristic against what actually
came next. On a fair feed it converges to 10%. If a genuine edge ever existed,
that panel would be the first place it appeared.
